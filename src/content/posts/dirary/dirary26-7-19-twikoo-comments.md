## 评论区终于开通了

今天给博客正式接入了评论系统。

以前博客更像是一个单向发布内容的地方：文章写完、部署上线，然后就结束了。读者即使有补充、疑问或者不同意见，也缺少一个直接交流的位置。现在文章页、留言板、友链页和赞助页都已经能够发表评论，博客终于多了一条从读者回到作者的通道。

这次采用的是 **Twikoo + Cloudflare Workers + D1**。网站本身已经部署在 Cloudflare，因此评论后端也继续放在同一个平台上，减少额外账号和服务之间的切换。

## 当前部署结构

评论系统由三个相互独立的部分组成：

| 部分 | 作用 |
| --- | --- |
| Firefly | 负责展示文章、评论输入框和评论列表 |
| Twikoo Cloudflare Worker | 接收评论请求，处理评论管理和接口逻辑 |
| Cloudflare D1 | 保存评论内容和 Twikoo 配置 |

评论后端使用单独的 Worker，数据库也与网站原有的排行榜数据库分开。这样评论功能出现问题时，不会影响网站静态页面和游戏排行榜。

Firefly 原本已经内置了 Twikoo 组件和适配样式，所以前端不需要从头开发。真正需要完成的是部署自己的 Twikoo 后端，再把后端地址写入评论配置。

## 接入过程

这次接入大致经历了以下步骤：

1. 部署 Twikoo 的 Cloudflare Worker 版本；
2. 创建独立的 D1 数据库；
3. 导入 Twikoo 所需的数据表结构；
4. 验证评论后端能够正常响应；
5. 将 Firefly 的 `envId` 改为自己的 Worker 地址；
6. 重新构建并部署网站；
7. 在评论区首次打开管理面板，设置管理员密码；
8. 发表测试评论，确认评论能够写入和读取。

一开始还考虑过使用 R2 保存评论中的图片，但当前需求只是文字交流。为了减少配置项和维护成本，最后没有启用 R2，先把纯文字评论做好。

## 已经支持的功能

目前评论区支持：

- 在不同文章下分别发表评论；
- 回复其他访客的评论；
- 评论点赞；
- 使用昵称、邮箱和个人网址作为访客信息；
- Markdown、代码块和表情等 Twikoo 基础输入能力；
- 在管理面板中查看、隐藏和删除评论；
- 文章、留言板、友链页和赞助页使用各自独立的评论线程；
- 评论数据保存在自己的 Cloudflare D1 中。

邮箱等敏感信息不会直接展示给普通访客，但站长可以在管理后台查看，用于识别评论和后续管理。

## 当前不足

### 1. 访客信息没有经过邮箱验证

当前的昵称和邮箱只是访客填写的评论信息，并不等同于注册账号。

发表评论前不会向邮箱发送验证码，因此系统不能证明填写者真正拥有这个邮箱。只要格式符合要求，访客就可以填写任意昵称和邮箱。

### 2. 不能保证一人一个身份

Twikoo 默认是匿名评论系统，而不是完整的用户系统，因此目前存在以下情况：

- 同一个人可以更换昵称或邮箱发表评论；
- 同一个邮箱可以搭配不同昵称；
- 不同的人也可能填写相同的昵称或邮箱；
- 浏览器记住访客信息只是本地保存，不代表已经登录。

所以评论旁边显示的昵称只能表示“这条评论填写了这个名字”，不能作为严格的身份认证。

### 3. 暂时没有邮件通知

当前 Cloudflare Worker 版本没有配置邮件服务。有人发表评论或回复时，系统不会自动给站长或访客发送邮件。

即使以后接入 SendGrid 一类邮件服务，普通的“评论提醒邮件”和“邮箱所有权验证”也是两回事。前者只负责通知，后者需要验证码、登录会话和唯一用户记录。

### 4. 不支持评论图片上传

这次没有启用 Cloudflare R2，因此评论区定位为纯文字交流。这样功能更简单，也不需要考虑图片存储、公开访问域名和清理策略。

### 5. 仍然需要处理垃圾评论

公开评论区不可避免地会遇到广告、重复内容和恶意提交。Twikoo 支持人工审核、反垃圾服务和 IP 频率限制，但这些能力需要根据实际情况继续配置。

## 为什么暂时接受这些不足

对个人博客来说，现在更重要的是先建立一个稳定、简单、可维护的交流入口，而不是立即开发一套完整的账号系统。

如果一开始就加入邮箱验证码、用户注册、密码找回、会话管理和账号注销，评论功能会迅速变成另一个独立项目。它不仅需要更多代码，还需要长期维护邮件送达率、账号安全和隐私规则。

目前的匿名评论模式虽然不能严格确认身份，但使用门槛低。访客不需要注册账号就能留言，这也符合个人博客轻量交流的定位。

## 后续可能的改进

如果以后评论数量增加，可以逐步考虑：

1. 开启评论人工审核或敏感内容过滤；
2. 配置 Akismet 等反垃圾服务；
3. 调整单个 IP 在一定时间内的评论次数；
4. 通过 SendGrid 增加评论和回复通知；
5. 如果确实需要唯一身份，再评估 GitHub 登录或邮箱验证码系统；
6. 定期备份 D1 中的评论数据。

如果未来更看重“每个人只有一个可验证身份”，可以切换到要求 GitHub 登录的 Giscus，或者自行开发邮箱验证码登录。不过这两种方案都会提高访客发表评论的门槛，需要在可信身份和使用便利之间做取舍。

## 小结

这次评论区接入的目标不是一次性实现所有功能，而是先完成一个清晰的最小闭环：

```text
访客发表评论
    ↓
Twikoo Worker 处理请求
    ↓
Cloudflare D1 保存评论
    ↓
文章页面重新读取并展示
    ↓
站长通过管理面板处理评论
```

现在这个闭环已经跑通。它并不完美，但评论数据掌握在自己手中，部署结构也相对简单。接下来可以先观察真实使用情况，再决定哪些不足值得投入时间解决。
