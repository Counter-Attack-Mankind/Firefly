使用 ChatGPT 和 Codex 一段时间后，我越来越觉得：决定结果质量的，不只是模型“聪不聪明”，还包括我怎样描述问题、给了什么上下文、允许它使用哪些工具，以及最后怎样验证结果。

AI 可以迅速生成一份看起来完整的答案，但“完整”不等于“正确”，“能运行”也不等于“满足真实需求”。真正有效的使用方式，是把 AI 当作一个能力很强、速度很快，但仍然需要边界、证据和审查的协作者。

## 1. Prompt、Skills、Agent 与 Harness

这几个概念经常同时出现，但它们处在不同层次。

| 概念 | 可以怎样理解 | 主要解决的问题 |
| --- | --- | --- |
| Prompt | 当前任务的需求说明书 | 这一次要做什么 |
| Skills | 可复用的专业工作流程 | 这一类任务应该怎样做 |
| Agent | 能分析、调用工具、执行并反馈的行动者 | 谁来完成任务并作出下一步判断 |
| Harness | 包围 Agent 的执行与约束系统 | Agent 在什么环境中、按什么规则完成任务 |

### 1.1 Prompt：给模型一份可检查的任务说明

Prompt 不只是“问一句话”，而是给模型定义目标、上下文、输出和边界。

一个实用的代码 Prompt 通常包含：

- **目标**：最终希望产生什么结果；
- **上下文**：相关文件、现有行为、复现步骤和技术栈；
- **约束**：哪些接口不能改，哪些文件不能碰，是否允许安装依赖；
- **验收标准**：怎样才算完成，要运行哪些测试；
- **输出要求**：需要直接修改代码，还是先分析、评审或给方案。

OpenAI 的提示词文档也把任务信息概括为目标、上下文、输出和边界。关键不是写得越长越好，而是把**会改变结果的信息**说清楚。[OpenAI Prompting 文档](https://learn.chatgpt.com/docs/prompting)

例如，“帮我修复保存功能”过于模糊；下面的描述更容易得到可验证的结果：

```text
问题：设置页点击“保存”后提示成功，但刷新页面后开关恢复原值。

复现步骤：
1. 打开 /settings。
2. 修改 Enable alerts。
3. 点击保存并刷新页面。

约束：
- 不修改现有 API 结构。
- 保持改动范围尽量小。
- 修复后增加回归测试，并重新执行上述复现步骤。
```

### 1.2 Skills：把重复经验变成可复用流程

Skill 可以理解为一份针对某类任务的操作手册。它不只是几句提示词，还可以包含：

- `SKILL.md` 中的触发条件与执行步骤；
- 辅助脚本；
- 参考资料；
- 模板和其他资源。

例如，处理 Word 文档时，可以规定“读取内容、修改、渲染为图片、检查版式、发现问题后继续迭代”；做代码审查时，可以规定“先报告高风险缺陷，再报告测试缺口，最后才做总结”。

Prompt 是一次性的任务，Skill 是可以反复调用的方法。官方文档也将 Skills 定义为可复用工作流和专业知识的载体，并采用按需加载，避免每次都把所有说明塞进上下文。[Codex Skills 与定制文档](https://learn.chatgpt.com/docs/customization/overview#skills)

如果一条规则对仓库里的每次任务都成立，例如测试命令、代码风格和审查要求，更适合写进 `AGENTS.md`；如果它是一套跨任务复用的完整流程，则更适合做成 Skill。

### 1.3 Agent：不只回答，而是持续行动

普通问答通常是“输入问题，输出答案”。Agent 更接近一个循环：

```text
理解目标 → 观察环境 → 选择工具 → 执行动作
    ↑                              ↓
根据结果调整 ← 检查输出与当前状态 ← 获取反馈
```

因此，Agent 不只是模型本身。它还需要目标、上下文、工具和停止条件。例如 Codex 可以读取仓库、搜索调用关系、编辑文件、运行测试，再根据报错继续修复。

Agent 的价值在于处理多步骤任务；它的风险也来自同一点：如果一开始误解目标，它可能非常勤奋地沿错误方向走很远。所以 Agent 越自主，越需要明确边界、阶段检查和真实验证。

### 1.4 Harness：让 Agent 真正工作的支撑系统

Harness 不是我这里指代的某个单独按钮或固定产品功能，而是一个工程概念：**包围模型并让 Agent 能稳定执行任务的整套支撑结构。**

它通常包括：

- 上下文如何被收集和压缩；
- 可以调用哪些工具；
- 文件、网络和命令有哪些权限；
- 工具调用失败后怎样重试或升级权限；
- 如何记录过程、运行测试和评估结果；
- 什么时候暂停确认，什么时候判定任务完成。

同一个模型放进不同 Harness，表现可能完全不同。只给聊天框时，它只能提出建议；接入仓库、终端、测试、浏览器和权限系统后，它才可能完成“阅读、修改、运行、观察、再修改”的闭环。

可以用一句话概括四者的关系：

```text
Prompt 定义本次目标，Skill 提供可复用方法，
Agent 负责判断和行动，Harness 提供工具、环境、权限与验证闭环。
```

## 2. 真正写项目时，要回到第一性原理

AI 很擅长识别常见模式，也很喜欢调用熟悉的模板。多数时候这是效率优势，但它也可能把一个具体问题，过早地归入某个常见问题模板。

第一性原理要求先暂时放下“通常大家怎么做”，重新确认最基本的事实：

1. 用户真正想解决的问题是什么？
2. 系统必须保持不变的性质是什么？
3. 当前有哪些事实已经由代码、日志或实验确认？
4. 哪些只是我和 AI 的猜测？
5. 能解决问题的最小改动是什么？

例如程序运行缓慢时，模板化回答可能马上建议缓存、并行或更换算法。但第一性原理会先问：时间究竟消耗在哪个阶段？瓶颈是计算、磁盘、网络，还是锁等待？如果没有测量，所谓“优化”很可能只是在修改看起来可疑的代码。

所以，在让 AI 实施方案前，可以先要求它给出一条证据链：

```text
现象 → 可复现步骤 → 观测证据 → 根因假设 → 验证实验 → 最小修复
```

只要中间某一环缺失，就应该把结论暂时标为假设，而不是事实。

## 3. 对抗性审查：不要只问“完成了吗”

AI 的回答有时会表现得像“报喜不报忧”：强调已经完成的部分，弱化未验证的风险，或者把“代码已生成”写成“功能已实现”。这不必理解为它具有刻意隐瞒的主观动机，更可能是因为模型倾向于给出连贯、满足请求和具有完成感的回答。

无论原因是什么，使用者都不能把自信的语气当成证据。

### 3.1 常见的乐观偏差

- 只运行了编译检查，却声称功能已经验证；
- 只测试正常输入，没有测试边界和失败路径；
- 根据文件名或函数名推测行为，没有追踪真实调用链；
- 新增了代码，但没有确认代码是否被实际入口调用；
- 使用熟悉的架构模板，却没有证明项目真的需要这些抽象；
- 忽略未完成项，在总结中只罗列成功修改。

### 3.2 对抗性审查不是故意唱反调

它不是要求 AI 无休止地怀疑一切，而是主动寻找“足以推翻当前结论的证据”。

可以从四个角度审查：

- **反例**：什么输入会让方案失败？
- **边界**：空值、最大值、并发、超时和部分失败怎样处理？
- **回归**：修复当前问题时，可能破坏哪些已有行为？
- **证据**：结论来自测试、日志、源码，还是仅仅来自推测？

真正可靠的完成报告应该区分：

```text
已实现：代码已经修改。
已验证：具体测试或复现步骤已经通过。
未验证：因为环境、权限或数据限制没有检查。
剩余风险：即使测试通过，仍有哪些可能失败的地方。
```

## 4. 适合放在代码需求后的提示词

这些提示词不是每次都要全部使用。任务越大、代价越高、需求越模糊，越应该增加澄清和审查步骤；只改一个明确的错别字时，反复确认反而会降低效率。

### 4.1 需求复述与完整性检查

适合架构调整、跨模块功能、数据迁移和需求尚未完全确定的任务：

```text
在生成或修改代码前，请先完成需求审计：

1. 用你自己的话复述我的目标、范围和验收标准。
2. 列出你认为已经明确的信息。
3. 指出所有模糊、缺失或相互冲突的信息。
4. 说明你准备采用的关键假设，以及这些假设可能造成的影响。
5. 如果缺失信息会显著改变实现方案，请先提问并等待我确认；
   如果不影响主要方向，可以明确假设后继续。

在获得确认前，不要生成最终实现。
```

这里最好保留“只有显著影响方案时才等待确认”。否则 Agent 可能把每一个小细节都变成阻塞问题。

### 4.2 第一性原理检查

适合性能问题、复杂 Bug 和容易套模板的需求：

```text
请先从第一性原理分析这个问题：

1. 真正需要解决的核心问题是什么？
2. 哪些约束和现有行为必须保持不变？
3. 哪些结论已经有代码、日志或测试支持？
4. 哪些内容仍然只是假设？
5. 能解决核心问题的最小改动是什么？

不要因为它看起来像某个常见问题，就直接套用惯用方案。
```

### 4.3 暴露不确定性与遗漏

这两句话非常适合放在方案评审、根因分析或准备实施之前：

```text
第一，你眼下最没有把握的事情是什么？
请说明不确定性的原因，以及需要什么证据才能确认。

第二，关于当前情况，我最大的遗漏是什么？
有哪些我没有提到、没有意识到，但可能改变结论或实现方案的因素？
```

比起只问“还有问题吗”，这种问法会迫使回答把注意力放到最薄弱的环节。

### 4.4 完成前的对抗性审查

适合加在代码实现要求的最后：

```text
实现完成后，请不要立刻给出成功总结。先以反方审查者的身份检查：

1. 尝试找出至少三个可能失败的场景或反例。
2. 检查边界条件、错误路径、并发或状态恢复问题。
3. 检查是否套用了不适合当前项目的模板或抽象。
4. 区分“已实现”“已验证”“未验证”和“剩余风险”。
5. 运行与改动范围相匹配的最小测试，并报告真实命令与结果。

如果没有证据，请使用“推测”或“尚未验证”，不要写成确定结论。
```

官方 Codex 工作流同样强调复现步骤、约束和验证：修复后重新执行复现流程，并运行与改动相关的检查，而不是只查看生成的代码。[Codex Prompting 与验证示例](https://learn.chatgpt.com/docs/prompting#fix-a-bug)

## 5. 一份可以直接复用的组合模板

对于中大型代码任务，可以把前面的内容压缩成下面这段：

```text
在开始实现前：

1. 复述我的需求、范围、约束和验收标准。
2. 指出信息是否完整，以及模糊、缺失或冲突之处。
3. 说明你最没有把握的事情，以及需要什么证据才能确认。
4. 指出我可能遗漏、但会影响方案的重要因素。
5. 从第一性原理给出最小可行方案，不要直接套用常见模板。

若关键缺失会改变架构、数据或对外行为，请等待我确认；否则明确假设并继续。

实现完成后：

1. 进行一次对抗性审查，主动寻找反例、边界问题和回归风险。
2. 运行与改动相匹配的测试或复现步骤。
3. 分别列出已实现、已验证、未验证和剩余风险。
4. 不要用自信语气替代证据，也不要隐藏失败的命令或不完整结果。
```

## 6. 最后的感悟

Prompt 可以改善一次沟通，Skill 可以沉淀一套方法，Agent 可以把方法执行到底，Harness 则决定整个执行过程是否拥有足够的工具、约束和反馈。

但这些技术都不能替代人的判断。AI 最容易让人放松警惕的时刻，往往不是它明显答错的时候，而是它给出了一份结构完整、语言自信、看起来非常像正确答案的结果。

所以，使用 Codex 和 ChatGPT 的关键，不只是更会“提要求”，还要更会：

- 追问依据；
- 暴露假设；
- 寻找反例；
- 检查真实输出；
- 把重复教训写进长期规则和 Skills；
- 始终用测试和事实完成最后一次判断。

AI 可以承担大量思考和执行工作，但“什么算正确、什么风险可以接受”，最终仍然需要由人来决定。
